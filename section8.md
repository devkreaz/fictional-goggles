# Розділ 8. Обґрунтування вибору типу бази даних

## Контекст
Для розробки інформаційної системи керування бібліотекою необхідно обрати тип бази даних, який забезпечить ефективне збереження інформації про книги, користувачів, історію видачі та інші сутності.

## Варіанти типів БД

1. **Реляційна база даних (SQL)**
   - Переваги: структура даних чітко визначена, підтримка складних запитів, транзакції, зв'язки між таблицями.
   - Приклади: PostgreSQL, MySQL, SQLite.
   - Підходить для систем з формалізованими даними й необхідністю забезпечувати цілісність через зв'язки.

2. **Документоорієнтована NoSQL база**
   - Переваги: гнучкість схеми, зберігання JSON-подібних документів, легке масштабування за допомогою шардінгу.
   - Приклади: MongoDB, CouchDB.
   - Зручно, коли структура даних може змінюватись або коли потрібно швидко розгортати прототип.

3. **Ключ-значення та інші NoSQL**
   - Підходять для кешування чи простої інформації без складних зв'язків.
   - Приклад: Redis, Cassandra.

## Обраний тип
Для даної системи рекомендується використати реляційну базу даних (наприклад, PostgreSQL).

### Обґрунтування
- **Структура даних**: в бібліотечній системі є чітко визначені сутності (книги, читачі, видання, записи про видачу), які зручно моделювати в таблицях з нормалізованими зв'язками.
- **Транзакції**: при видачі/поверненні книг потрібно зберігати узгоджений стан (оновлення кількості на складі, запис в історії), що вимагає підтримки ACID.
- **Універсальність**: SQL дозволяє виконувати складні запити для звітів (наприклад, знайти книги за автором, перевірити прострочення).
- **Екосистема**: реляційні СУБД мають розвинуті інструменти резервного копіювання, відновлення, відладку перформансу.
- **Досвід розробників**: знання SQL широко поширене, що спрощує підтримку.

## Альтернативи
Якщо в майбутньому виникне потреба в горизонтальному масштабуванні при дуже великій кількості користувачів, можна розглядати гібридне рішення з використанням кешу (Redis) та/або документної бази для зберігання історичних записів.

## Висновок
Для початкового етапу розробки найбільш адекватним є використання реляційної СУБД. Вона забезпечує необхідну цілісність, продуктивність та простоту формування звітів, а також дозволяє еволюціонувати систему з мінімальними ризиками.
